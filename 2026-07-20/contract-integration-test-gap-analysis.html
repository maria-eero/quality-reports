Two snapshots in a row confirm this is systemic: Every sprint produces bugs from the same three failure categories (API contract violations, data sync / state mismatch, interface parity / schema drift). These represent gaps in the testing pyramid that allow known failure modes to ship repeatedly. Contract tests would prevent the first category; integration tests address the second and third. The PoC branches for both platforms demonstrate the pattern working today — see the Proof of Concept tab.

What We Found

No Contract Tests on Any Platform

  • ✗ Cloud - No Pact, no schema validation. Scala monolith has no visible test infra
  • ✗ iOS - No contract tests. Has 29 JSON fixtures + decoding tests, but these only verify "doesn't crash" — not that required fields are non-null or enum values are complete
  • ✗ Android - No contract tests. Has ~300 JSON fixtures, property-based round-trip tests, and deserializer tests — but none validate against the live API or enforce field-level contracts
  • ✗ Cross-platform - No shared schema. iOS and Android interpret API responses independently

Existing Tests Cover Happy Path Only

  • ! EeroRRM: Sibling activities test null baseMac, but IngestRrmData was skipped
  • ! iOS LiveDataUsage: Tests always pass non-zero data. Nil case never exercised
  • ! Android ContentFilters: Tests hardcode correct names in fixtures, pass even if API changes
  • ! DARS: Integration test package had only SimpleEchoTest before bug was found
Recommendation

Phase 1: Contract Tests (2-4 weeks)

  • ✓ Enum completeness (no throws on valid values)
  • ✓ Required field non-null validation at boundaries
  • ✓ Read/write key format consistency
  • ✓ Error code mapping (4xx vs 5xx)
  • ✓ Input validation before Kafka/DynamoDB writes

Phase 2: Integration Tests (4-8 weeks)

  • ✓ State transition completion (PENDING -> DONE)
  • ✓ Cross-table data consistency (SSID configs)
  • ✓ UI <-> device state sync after mutations
  • ✓ Override propagation through gRPC paths
  • ✓ Platform response interpretation parity

Tooling: Extend What Already Exists
Key insight: We are not starting from zero. iOS already has 29 JSON fixtures + decoding tests. Android has ~300 JSON fixtures + property-based round-trip tests. The gap is not "no test infrastructure" — it's "existing tests verify decoding doesn't crash, but never assert the specific failure modes causing production bugs."

Phase 1: Client-Side Schema Validation (no backend dependency)

  • iOS: Extend existing UserAPIDecodingTests — add XCTAssertNotNil for required fields (hw_model, baseMac) on the 29 existing fixtures
  • Android: Add Kotest schema assertions using existing io.kotest:kotest-property dependency — validate fixtures against defined field requirements
  • No new dependencies: Both platforms already have the test frameworks; we add assertions, not libraries
  • Drift detection: Add a scheduled CI job that refreshes fixtures from staging and alerts if schema breaks
  • Graduation to Pact: If cloud team buys in later, schemas become consumer contract definitions

Phase 2: Integration Tests

  • Target: State transition flows (DARS status), cross-table consistency (SSID configs)
  • Environment: Run against staging (real services, test data)
  • CI placement: Post-merge on staging deploy — alerts but does not block initially
  • Graduation: Once stable for 2 weeks, promote to merge-blocking
  • Framework: Lightweight pytest suite with API client (no UI dependency)
Ownership & Maintenance

Who Writes What

  • Consumer contracts (iOS/Android): QAE writes initial, platform dev maintains
  • Provider verification (Cloud): Cloud dev owns, QAE provides expectations
  • Integration tests: QAE writes initial suite, service team maintains alongside service

Maintenance Cost

  • Contract tests: Very low — test shape, not values. Break only on real schema changes.
  • Integration tests: ~0.5 days/quarter/suite for test data and env issues
  • Total ongoing: ~1-2 days/quarter once established (amortized across all suites)

What Blocks Merge

  • Day 1: Contract test failures block the specific consumer or provider PR
  • Week 4: Integration test failures alert Slack channel, do not block
  • Week 8+: Stable integration tests promoted to merge-blocking
