How Tests Run

Android — opt-in test:contract CI job

Contract and integration tests are excluded from the default per-PR unit-test job, and re-enabled only when a PR is labelled run-contract-tests. Even then, they run only for the packages whose owned files actually changed in the diff. Zero-diff PRs and unrelated PRs that carry the label produce a no-op job that exits early.

Default per-PR run

./gradlew testEeroDogfoodDebugUnitTest \
  -PtestShard=$CI_NODE_INDEX \
  -PtestTotalShards=$CI_NODE_TOTAL \
  -PexcludeContractTests=true

-PexcludeContractTests=true filters out any test file under /com/eero/android/contract/ or /com/eero/android/integration/ from the shard filter in build.gradle. Local unsharded runs are unaffected.

Opt-in test:contract job

rules:
  - if: '$CI_MERGE_REQUEST_LABELS =~ /(^|,)run-contract-tests(,|$)/'
script:
  - selected=$(./scripts/select-contract-tests.sh)
  - if [ -z "$selected" ]; then
      echo "No contract tests for this diff"
      exit 0
    fi
  - eval "./gradlew :app:testEeroDogfoodDebugUnitTest $selected"

Job source: .test-include.yml (commit 1a5d323). 30-minute timeout, publishes JUnit reports as artifacts on failure.

Diff-scoped selection

scripts/select-contract-tests.sh reads the PR diff against $CI_MERGE_REQUEST_DIFF_BASE_SHA (or origin/development locally), matches each changed file against an inline mapping table, and emits deduplicated Gradle --tests flags. Emits nothing when no mapped path changed, so the job exits 0 without touching Gradle.

Android — fixture flow

All CT-* scenarios share a single consolidated fixture:

app/src/test/resources/fixtures/port_security/scenarios.json
  └── "scenario_01_not_capable":            { ... GET /networks/{id}/connections snippet ... }
      "scenario_02_capable_off_connected":  { ... }
      "scenario_03_capable_off_disconnected": { ... }
      "scenario_06_active": { ... }
      "scenario_07_blocked_non_eero": { ... }
      "scenario_08_non_capable_eero": { ... }   ← intentionally identical to scenario 7
      "scenario_09_peer_propagated": { ... }
      "scenario_10_cloud_throttle": { ... }
      "scenario_11_enable_success", "scenario_12_disable_success",
      "scenario_13_enable_error", "scenario_14_network_poll_blocked": { ... }

PortSecurityFixtures.payloadOf(scenarioId) loads the entry, strips the description metadata, and returns the raw JsonObject. Tests then parse it through GsonFactory.create() — the same Gson instance production uses (with RuntimeTypeAdapterFactory wired for the sealed PortConnectionStatus class). Using a plain Gson() would throw JsonIOException on the sealed base type in modern Gson versions, so contract tests are validating the real decode path, not a simplified one.

iOS — Swift Testing bundle

iOS tests live in Eero/EeroNetworking/Tests/Port Security Tests/PortSecurityContractTests.swift and run inside the existing EeroNetworkingTests target. Fixtures are loaded from a single PortSecurity.json next to the test file and decoded via EeroAPI.jsonDecoder — the same JSONDecoder production uses. Invocation:

mise run test unitTests  # or the EeroNetworkingTests scheme in Xcode

No CI-config change needed — the tests run as part of the existing PR test job on ios-client. Once the branch is rebased and PR'd, the tests execute on every PR that touches Eero/EeroNetworking/.

What a failure looks like

When a contract violation lands (fixture regenerated from staging, or a model field drift), the failing scenario surfaces in the CI report exactly as any other unit-test failure — but with a scenario ID and the tech-spec matrix row in the assertion label:

FAILED com.eero.android.contract.portsecurity.PortSecurityContractTest
  > CT-007 security on, non-eero detected parses as BLOCKED with data disabled

    expected: BLOCKED but was: ACTIVE
    at PortSecurityContractTest.kt:70

The failure names the CT-id from the tech-spec scenario matrix, the semantic assertion that broke, and the line — so a reviewer can jump straight from the failing PR into the exact scenario being violated. On iOS the same convention applies via Swift Testing's @Test("CT-007: …") label.

How It Works

The Pattern: Golden Fixture + Scenario Matrix

Each platform has a single JSON fixture file containing every documented API scenario. Tests load the fixture, parse it with the production decoder, and assert the model matches expected values.

// iOS — one test from the suite:
@Test("CT-007: Security on, non-eero detected — BLOCKED, data disabled")
func ct007_blockedNonEero() throws {
    let connections = try loadConnections()          // loads PortSecurity.json
    let port = try interface(7, in: connections)     // scenario on interface 7
    let security = try #require(port.portSecurity)

    #expect(security.capable == true)
    #expect(security.enabled == true)
    #expect(security.status == .blocked)
    #expect(port.disabledProtocols.contains(.data))
}

// Android — same scenario:
@Test
fun `CT-007 security on, non-eero detected parses as BLOCKED with data disabled`() {
    val port = firstPortOf("scenario_07_blocked_non_eero")

    assertEquals(true, port.portSecurity?.capable)
    assertEquals(true, port.portSecurity?.enabled)
    assertEquals(PortSecurityStatus.BLOCKED, port.portSecurity?.status)
    assertTrue(port.disabledProtocols.orEmpty().contains(PortProtocol.DATA))
}

Why This Instead of Pact

This Approach (Golden Fixture)Pact (Consumer-Driven Contracts)
Backend team involvementNone — fully client-sideRequired — provider must run verification
New infrastructureNone — runs in existing test suitePact Broker + provider CI job
New dependenciesZeroPact-Swift / Pact-JVM + broker client
Time to first valueDays (write fixture + assertions)Weeks (broker setup, provider onboarding)
What it catchesDecoder breaks on unexpected API shapesSame + bilateral deploy-time gating
What it doesn't catchSilent API drift (until fixture is refreshed)Nothing (full contract enforcement)
MaintenanceUpdate fixture when API intentionally changesNegotiate contract changes with broker
Graduation pathFixtures → Pact consumer expectations (low effort)Already at full maturity
Trade-off summary: Pact is stronger (bilateral enforcement, deploy gating) but requires backend participation we don't have today. The golden-fixture approach gives us 80% of the value with 0% of the dependency — and the fixtures become Pact consumer expectations if we graduate later.

What This Catches That Doesn't Exist Today

Toggle State Mismatch

If the port-security toggle emits before device state resolves, INT-002/003 fail — the ViewModel's emitted state must reflect the polled backend state, not stale cached values.

Directly prevents CORE-31743 pattern (July snapshot)

New Enum Value Crash

CT-M4 asserts that "FUTURE_STATE" decodes without crashing. If someone changes the enum to a failable init, this test catches it.

Directly prevents CORE-31436 pattern

Schema Drift

If a dev accidentally changes the model in a way that breaks parsing of the documented scenarios, the CI test fails before merge.

Why the mapping matters: Each of the CORE tickets on the right side of this table represents a real bug that shipped. The tests on the left side aren't "extra insurance for a hypothetical failure" — they're the specific assertions that would have caught the specific line of code that shipped broken. That's why the PoC targets the highest-volume Sentry offenders and the July P0s: same class of failure, tests catch it.
